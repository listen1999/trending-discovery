**[关键约束]**

**深度综述**
这条信号最有意思的地方不是某个单一数字，而是收入结构的迁移方向：Roberto Blake 的自述清楚地把"被动联盟佣金"排在增速最慢、最不可控的位置——TubeBuddy 改了佣金结构、终止了终身返佣，只留下一张 $13/月的"买断"支票；相比之下，教练/服务类"主动收入"虽然耗时间，却是他自己认定终身价值最高的部分。这和很多人对"被动收入"的浪漫化想象是相反的：真正跑了十年的创作者反而在往更主动、更可控的收入结构上收缩，而不是相反。风险与局限也很明显：他的现金流建立在 60 万订阅、十年内容资产和 800+ 辅导客户案例基础上，这是没法在短期内复制的存量壁垒，国内创作者如果只学到"要多元化收入"这一层，很容易高估自己现阶段能复制的部分——刚起步的公众号/小红书账号既没有对标的联盟体系，也没有足够体量支撑教练定价。更值得参考的是"工具化"这一步的时点选择：他不是一开始就做工具，而是先在自己的教练客户内部验证了一段时间（"For the past few months this year I've been using my own custom software tool with my coaching clients"），确认好用之后才开放对外销售——这是一条比"先做工具再找客户"风险更低的路径，值得档位 D 读者参考这个"先内部验证、再外部开放"的节奏，而不是照抄营收数字本身。

### 金矿 3：Codex/Claude Code 长会话失忆解法 — HANDOFF 三件套，可直接复制的提示词框架

来源：@runes_leo（引用 @KyrieCheungYep）· 2026-07-19 11:38 · 👍31 👁6,566 🔖44 · engagement_rate 0.67%（高于本期中位数 0.12%）
内容类型：Thread 引用还原（原贴 + 转发者补充踩坑经验）

**完整步骤（逐条列出，不压缩，来自 @KyrieCheungYep 原贴）**
1. **HANDOFF.md｜跨天任务必备**。长会话要结束、跨天推进的大任务、项目中间状态复杂时使用。复制提示词："这个会话要结束了。请写一份交接文档存到 `HANDOFF.md`：我们在做什么任务、已经完成了什么、当前卡在哪、下一步计划是什么、有哪些踩过的坑绝对不要再踩。写给一个完全没有上下文的新会话看。"下次开新会话第一句："请先读取 `HANDOFF.md`，了解项目上下文，再继续推进。"
2. **反思机制｜把踩过的坑变成下一轮输入**。复制提示词："回顾我们这次协作，你的输出里，哪些内容是我给你纠正的，逐条列出，判断是我给的信息不够导致你判断缺失，还是信息够了但你的判断逻辑有问题，思考如果下次做同样的任务，我在开头多说哪几句话，你就能避开这些问题？复盘结果用表格输出三列，包括我修改的内容、归因、下次的开头指令建议。"
3. **经验记录｜把纠错变成长期资产**。复制提示词："把我们刚才的经验录入错题本/系统记忆，只记纠错与有效方法，错题本里已有类似记录的，更新旧条目，不新建，审查规则或提示词模板里已经明确写了的内容，不重复记录。"
4. 使用顺序：长会话收尾先写 HANDOFF.md（只记进度，不堆经验）→ 中间发现偏差跑反思机制（存进错题本）→ 错题本攒出高频规律后提炼进 Memory 常驻

**@runes_leo 的补充踩坑经验**：HANDOFF 共享给多路代理后，任务/进度/卡点/下一步/踩坑都写清了，各路仍会各自推进、判断漂移——缺的不是信息，是"锁"：谁有权改结论、什么证据才允许改、哪一关必须人点头。他给出的三行补丁：当前唯一负责人 / 改结论所需的新证据 / 必须人工确认的下一关。

**前置条件/适用人群**：使用 Codex、Claude Code 等 Agent 编程工具、且经常进行跨天/多轮长任务协作的档位 C 读者
**国内可用性**：直接访问——Codex、Claude Code 均为本方法论的载体工具，方法本身是提示词模板，不依赖特定平台，理论上可迁移到国内任何支持长上下文的 Agent 工具
**预计耗时**：单次应用不超过 5 分钟（复制粘贴提示词）

**web_search 核实结果**：经 web_search 核实，"HANDOFF.md / 会话交接文档"是当前 AI 编程工具生态里一个已被多方独立验证的通用模式，除 KyrieCheungYep 的实践外，还能查到 mcpmarket、skillstore、explainx.ai 等平台上有专门封装成"Session Handoff Skill"的产品化尝试，说明这不是孤例式的个人技巧，而是正在被沉淀为工具化能力的共识型方法论 [来源：jdhodges.com、mcpmarket.com、skillstore.io 等多篇独立文章]

**原始链接**：https://x.com/KyrieCheungYep/status/2077770749155414080

**深度综述**
这套方法论真正的价值不在"写交接文档"这个动作本身（这已经是社区共识），而在于 runes_leo 补充的那一层——多路代理并行时，信息完整不等于判断一致，缺的是"谁有权拍板"的权限锁，而不是更多的记录字段。这是一个挺反直觉的洞察：大多数人遇到 Agent 协作出问题，第一反应是"记录得不够细"，但 runes_leo 的实测经验指出，真正导致"各自推进、判断漂移"的原因是没有唯一负责人和明确的改结论门槛，这更接近团队管理里的决策权设计问题，而不是纯粹的工程记录问题。趋势定位上，这属于"中期验证"阶段的信号：HANDOFF 模式已经从个人小技巧演化出独立的 Skill 产品（如上文核实的 mcpmarket/skillstore 案例），意味着这套方法论正在被更多人验证并工具化，不再是一次性的个人心得。局限在于，这套提示词对"轻量单人任务"可能是过度设计——如果只是几十分钟能做完的小任务，写 HANDOFF.md 反而增加负担，它的收益在"多日、多代理、状态复杂"的场景里才会显现，档位 C 读者应该按任务复杂度分级使用，而不是所有会话都套用三件套。

## 避坑指南

- **静默失败比报错更危险**：runes_leo 的实测经验显示，自动化工作流如果只做"模型失败就换一条路重试"，而不显式标注"当前输出是否降级"，很容易拿到一份"看起来完整但实际不可信"的结果——尤其是依赖多模型 fallback 的 Agent 工作流，建议每次任务收尾都补上"实际用的模型/尝试结果/输出状态"三项标注。
