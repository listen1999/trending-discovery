### 金矿 1：VPS + Cloudflare + Tailscale + Claude Code——把服务器管理外包给 AI 的 16 步方案

来源：@levelsio（转推 @robj3d3）· 2026-07-23 06:11 · 👍1958 👁117,788 · 收藏 5,234
engagement_rate 4.44%（同期中位数约 0.15%-0.2%，本条为中位数的 20 余倍，本期最强信号）
内容类型：Thread（16 步骤，原文完整，无需外部展开）

完整步骤（逐条列出，不压缩）
1. 买一台便宜 VPS（Hetzner 或 DigitalOcean，约 €5-10/月），Ubuntu 24.04，购买时勾选自动备份（约15分钟）
2. 把域名接入 Cloudflare（免费版），在域名注册商处切换 DNS 服务器
3. 笔记本和手机装 Termius（SSH 客户端）+ Tailscale（私有组网），两者都有免费版
4. 在 Termius 里生成 SSH key，创建 VPS 时就加进去，只认密钥、不认密码
5. 用公网 IP SSH 进去一次，跑系统更新，装好 Tailscale 并登录
6. 关掉服务器 Tailscale key 的过期时间（管理台点一下）
7. 在做下一步之前，先确认能通过 Tailscale 分配的 100.x 地址 SSH 进去
8. VPS 服务商防火墙：删光所有入站规则，只放行 Cloudflare 官方 IP 段的 443 端口，不留公网 SSH，只能走隧道进
9. 从外部测试：公网 IP 全部超时，Tailscale IP 能连通，服务器对外"隐形"
10. 一台设备配一把 SSH key，手机单独一把加进 authorized_keys
11. `apt install tmux`，再装 Claude Code（官方原生安装器，一条 curl 命令）
12. 在 tmux 会话里跑 Claude Code，笔记本断开、合盖它也继续工作
13. 写一条长的交接 prompt 告诉 Claude Code：服务器现状、安全模型（别去"修"防火墙）、目录约定（每个项目 /srv/http/域名，各自一个 tmux 会话）、个人偏好、以及"破坏性操作前必须确认"这类硬规则
14. 让 Claude Code 把这些先写进 CLAUDE.md，后续每次会话都自带这份记忆
15. 先建好每晚推送数据到 GitHub 的备份任务并测试通过，再开始写第一个功能
16. 之后让它装 Web 服务器（Caddy + Cloudflare DNS 插件跟锁死的防火墙配合最顺）、配 SQLite、上线第一个页面

前置条件 / 适用人群：具备基础命令行/SSH 操作能力、愿意把服务器交给 Agent 管理的独立开发者（档位 B 最适用，档位 C 若已用惯 Claude Code 也可尝试）
国内可用性（分项，经 web_search 核实）：
- Hetzner / DigitalOcean：可注册使用，但服务器在境外，国内访问速度和稳定性依线路而定，且未做 ICP 备案不能用于面向中国大陆的公网服务
- Cloudflare 免费版：国内可直接访问，部分节点速度一般
- Termius：普通 SSH 客户端，可直接访问
- Tailscale：**需要工具**——经核实其协调服务器域名在国内部分省份线路上有被墙记录，底层 WireGuard 协议未做流量混淆，容易被 DPI 识别，是国内读者复现这套方案时最大的障碍
- Claude Code（Anthropic 官方渠道）：**需要工具**，中国大陆无法直接访问 claude.ai / Anthropic API，需科学上网或走中转 API
预计耗时：推文原文自述约 40 分钟（购买 15 分钟 + 加固 20 分钟 + 装 Agent 5 分钟）
可直接使用的代码 / 配置：无附带脚本，是可直接照抄执行的步骤清单
原始链接：https://x.com/levelsio/status/2080053133896523954

深度综述：
这条信号最有意思的地方不是"用 AI 写代码"，而是"把运维这件事本身外包给 Agent"——第 13-14 步的核心是把安全模型、目录规范写成一份 CLAUDE.md 记忆，让 Claude Code 在每次新会话里都不需要重新学习这台服务器的"个性"，这本质上是把运维知识从人的脑子搬进了可被 Agent 读取的文档里，这是过去"自己写 shell 脚本自动化"做不到的持久化能力。反直觉的地方在于，它并不追求"零人工干预"，第 13 步明确写了"确认后再执行破坏性操作"的硬规则——说明即便是重度 AI 自动化的实践者，也在有意识地给 Agent 设边界，而不是完全放权。风险与局限上，这套方案对中国读者最大的短板是 Tailscale：一旦协调服务器连不上，第 7-9 步的"隐形服务器"验证就无法完成，整套安全模型会卡住；比较现实的替代路径是自建 WireGuard 或用国内可用的跳板机方案替换 Tailscale 这一环，其余步骤（Cloudflare、Termius、Caddy、Claude Code 的记忆写法）基本可以照搬。竞争格局上，这类"VPS 自建 + Agent 托管"打法本质是在跟 Vercel/Railway 这类全托管 PaaS 抢一人开发者的心智，优势是成本极低（€5/月 vs 云 PaaS 动辄 $20+/月起）、缺点是需要读者自己承担安全配置的认知成本，这也是为什么第 4-10 步几乎全部在讲"锁服务器"而不是"跑代码"。

核心功能（聚焦对一人公司的价值）
拦截并可视化 Claude Code、Codex CLI、Gemini CLI、Cursor CLI 等十余种 Coding Agent 的真实 API 流量：完整 System Prompt、对话历史、Tool Schema、Tool Call / 结果、流式响应、Token 用量，以及相邻请求之间的结构化 Diff，最终生成一个零外部依赖的自包含 HTML 文件供本地回放。

定价
- 免费层：完全开源免费
- 付费层：无

10 分钟上手
步骤 1：clone 仓库并按 README 完成本地代理安装
步骤 2：把 Claude Code / Codex 等工具的请求出口指向 claude-tap 的本地代理端口
步骤 3：正常触发一次真实任务，在自动生成的 HTML/浏览器查看器里回放整条 Trace

与现有工具链配合：对工具集成者 / vibe coder（档位 C）来说，这是理解"上下文怎么组装、工具怎么注册、状态怎么跨轮传递"这类 Agent Harness Engineering 知识最快的路径，能直接对照自己在用的 Prompt / Skill 设计做调优；对独立开发者（档位 B）自建 Agent 类产品时，也可以用来复盘线上 Agent 的失败案例、定位 Tool Call 出错的具体环节。

踩坑预警 / 已知限制：需要一定的本地代理 / 网络配置基础；抓包内容可能包含敏感 Prompt 或密钥信息，本地留存要注意数据安全，不建议在共享设备上直接跑。

竞品对比：同类可观测性方案多是 LangSmith 这类云端 SaaS，claude-tap 的差异化在于完全本地化、零外部依赖，且直接面向 CLI 类 Coding Agent，而不是面向 LangChain 这类框架级调用。

官方链接：https://github.com/liaohch3/claude-tap

深度综述：
这条信号处于"Agent Harness Engineering"这个新兴技能被工具化的中期验证阶段——过去半年"怎么给 Agent 设计好用的 Prompt/Skill"更多停留在经验分享层面，claude-tap 这类工具把它变成了"可以对照源码和真实 Trace 学习"的具体实践，dotey 的转推文案里直接点出"理解运行时系统：上下文如何组装、工具如何注册"这个学习路径，这跟本期出现的另一条信号（vista8 的 Claude Skill）放在一起看，说明"把 Claude Code / Agent 当成可编程平台去研究"正在从少数极客行为变成一部分独立开发者的日常学习方式。意外之处在于，2.7k star 的体量说明这不是一个小众玩具，而是已经有相当规模的开发者在用它排查 Agent 行为问题，侧面反映出"Agent 出了问题但看不见黑盒里发生了什么"是当前 vibe coder 群体的真实痛点。风险与局限上，这类工具的价值高度依赖使用者本身有能力读懂 System Prompt 和 Tool Schema 的设计逻辑，对纯粹"会用不懂原理"的用户来说，直接收益有限，更适合已经在自建 Agent 产品、需要排查线上问题的档位 B/C 读者。

### 金矿 4：Senja 的 $1M ARR 战略十字路口——一份可复用的"要不要转型"决策清单

来源：@helloitsolly（Olly，Senja 联合创始人）· 2026-07-23 18:47 · 👍21 👁3,041 · 收藏 15
engagement_rate 0.49%（同期中位数约 0.15%-0.2%，高于中位数但绝对浏览量不高，属于面向同行小圈子的精准讨论）

核心数据（已验证）
- ARR：经 author_bio 及 web_search 交叉核实，Senja 官方博客与第三方案例站（IndieMerger）均确认其已达 $1,000,000 ARR（约合 ¥677 万，按 1 美元≈6.77 元人民币计算，汇率查询于 2026-07-21），MRR 约 $83,300，付费客户约 3,000
- 该数据为此前已披露的既有成绩，本条推文本身是围绕"下一步往哪走"的实时战略思考，不是新的收入披露

商业模式拆解 / 决策框架
Senja 是一款帮 SaaS 团队收集与展示客户证言（testimonials）的工具，推文里 Olly 公开列出了 5 个可能的转型方向并逐一给出利弊：
1. 继续深耕证言这个小而美的市场
2. 转向 Shopify 评论市场（比证言大 100 倍，但会从细分领先者变成大池子里的小鱼）
3. 转向"AI 生成客户故事素材"（Canva for customer stories）
4. 转向更泛化的"客户反馈"赛道（市场大 1000 倍但同质化严重）
5. 转向客户成功软件 / CRM（客单价更高但销售更重、竞品更重）
他给自己列的判断清单是：哪个方向能在现有产品基础上直接扩展、哪个方向是自己独有的分发优势、哪个方向已有搜索需求、哪个方向在 AI 时代更有护城河、以及 3000 个现有客户到底想要什么。

复制路径（只写真正适用的档位）
档位 D（服务变现者）：当自己的服务/工具遇到客单价或市场天花板时，可以直接照搬这份 checklist——"这个新方向能否复用现有客户和分发渠道"往往比"新市场是不是更大"更该优先判断，盲目奔向更大市场容易从细分头部变成大池子里的无名小鱼。
档位 B（独立开发者）：做垂直工具型 SaaS 到一定规模后普遍会遇到的"横向扩品类 vs 纵向做深"抉择，可以参考 Olly 公开列出的 5 个选项结构，作为自己评估潜在新方向的模板，而不是照抄具体选择的方向。

竞争格局
证言/评论工具赛道里 Shopify 评论方向已有 @judgeme 等增长很快的玩家，客户成功 / CRM 方向则是更重的赛道，Olly 自己也承认"我们会是大池子里的小鱼"，这也是他犹豫的核心原因。

深度综述：
这条信号的价值不在"又一个 SaaS 创始人赚了多少钱"，而在于把一个真实、尚未做出决定的战略十字路口原样公开——这在 Build in Public 的语境里其实少见，多数创始人只愿意分享"做对了什么"，Olly 直接把"我现在纠结什么、每个选项的坏处是什么"摊开写。反直觉的地方在于，他把"市场大 100 倍到 1000 倍"的选项排在了相对谨慎的位置，而不是默认"市场越大越该冲"，这跟很多增长叙事里"找更大赛道"的直觉建议相反，说明对一个已经有 3000 付费客户、$100 万 ARR 规模的团队来说，"能不能把现有客户和分发能力复用到新方向"比单纯的市场规模更重要。风险与局限上，这条信号最大的局限是它是一份未完成的思考，没有最终答案，不能直接当成"应该往哪转型"的结论去照抄；而且 Senja 的决策背景（3000 付费客户、成熟分发渠道、多年品牌积累）是很多刚起步的独立开发者不具备的存量，同样的选项框架放到一个刚上线 3 个月的产品上可能完全不适用，复制这条路径时最容易踩的坑就是照抄"往哪转"而不是照抄"怎么判断"。

## 避坑指南

- Tailscale 在国内不可靠：金矿 1 的自托管方案里，Tailscale 是唯一一个国内读者大概率会卡住的环节——经核实其协调服务器域名在部分省份线路上有被墙记录，WireGuard 协议本身也容易被识别限速。照搬这条教程之前，先确认自己有稳定方案打通这一环（自建 WireGuard、境内跳板机），否则第 7-9 步的"服务器隐形"验证根本跑不通。
- 研究结论正确不等于执行正确：runes_leo 的复盘（尽调组合理论收益 +81%，实际仓位却 -69%）提醒，用 AI 做研究判断和照研究结果去执行是两个独立的能力，AI 能放大产能，但不会自动带来客户和现金流，真正的门槛始终是"把判断稳定地兑现成结果"。
