### 金矿 1：Grok Build — 出事之后被迫开源的编程 Agent Harness

[Grok Build] | xAI 开源的终端编程 Agent Harness + TUI，因"打包上传整仓库"隐私丑闻紧急开源接受审查
来源：@vista8 · 2026-07-16 08:14（21.8h ago）· @xiaohu / @itsolelehmann / @lidangzzz 本期跟进转发解读
发布/更新日期：开源于 2026-07-15 前后（原始开源事件略早于本窗口，本期是社区拆解代码、star 数暴涨的窗口）
国内可用：需要工具（Grok/xAI 官方服务需科学上网及非中国大陆手机号注册；GitHub 访问在国内不稳定，但源码可自行镜像/编译后本地部署）

核心功能（聚焦对一人公司的价值）
- Rust 编写的全屏、鼠标交互 TUI，理解代码库、编辑文件、执行 shell 命令、联网搜索，是 Grok 4.5 编程能力背后的同一套 agent 运行时，约 100 万行 Rust 代码
- 现已支持完全本地优先：自行编译、接自己的本地推理模型、所有配置走 config.toml，理论上可脱离 xAI 云端运行

定价
- 免费层：整个 harness 开源，Apache 2.0 协议，自行编译免费；预编译二进制提供 macOS / Linux / Windows
- 付费层：若接入 Grok 官方 API，仍按 xAI 官方 token 计费（本期未查到最新费率，需以 x.ai 官网为准）[未经验证]

10 分钟上手
1. `git clone` 仓库（约 100 万行 Rust 源码）
2. 自行编译，或直接下载对应系统的预编译二进制
3. 编辑 config.toml 指向本地推理模型或自己的 API key，跑通后即可在终端内驱动整个编码流程

与现有工具链配合（具体场景）
- 可作为 Claude Code / Codex / Cursor 之外的第四个终端 agent 选项，适合已在多个 agent 间轮换以规避单一厂商限流/重置额度的重度用户——本期数据里 @xiaohu、@runes_leo 都提到"Claude 重置几分钟后 Codex 也重置"，多 agent 备份是海外开发者的普遍策略

踩坑预警 / 已知限制（本条核心）
- 安全研究者 cereblab 于 7 月 12 日公开的抓包分析显示：Grok Build 会把开发者本地仓库的完整 Git 历史（含已提交的密钥）打包上传到 xAI 自营的 Google Cloud Storage bucket，上传量约为任务实际需要代码量的 27,800 倍
- 更严重的是，CLI 里唯一的隐私开关"Improve the model"关闭后毫无作用——关闭后重新抓包，`/v1/settings` 接口仍返回 `trace_upload_enabled: true`，上传通道照常接收分片提交
- xAI 于 7 月 13 日加了服务端开关拦停上传，7 月 14 日晚发布 ZDR（零数据保留）政策和 /privacy 页面，xAI 高管 Andrew Milich 承诺可对所有用户做溯源删除，马斯克本人承诺清空此前所有已上传数据——但这些补救都发生在丑闻曝光之后，不是设计时就有的保障
- 若此前用 Grok Build 处理过客户项目代码或含密钥的仓库，理论上内容已被上传过，值得回头检查密钥是否需要轮换

竞品对比
- Claude Code / Codex CLI / Cursor CLI 目前没有被曝出类似"开关失效、整库外传"的问题，但本次事件提醒：任何 agent CLI 的"本地优先"承诺都需要自己抓包验证，不能只看营销页面

官方链接
- GitHub：https://github.com/xai-org/grok-build
- xAI 官方开源公告：https://x.ai/news/grok-build-open-source

深度综述
这条信号最反直觉的地方在于：一个被曝"关了隐私开关也没用"的工具，居然靠"开源"完成了公关自救，开源后一天内涨了近 1 万星（经 GitHub API 核实，目前已到 11,752 星、1,976 fork）。这说明开发者社区对"开源=可信"有近乎本能的信任——哪怕这份信任本身也需要打问号：开源的是运行时代码，不代表 xAI 服务端此前抓的数据会被公开审计，唯一能验证的只是"从现在起可以自己编译、接自己的模型、脱离 xAI 服务器"这件事。往前看，这是"AI coding agent 信任危机"系列信号里最新也最典型的一例——过去半年，从"Improve the model"开关到"local-first"宣传语，几乎每家 agent 厂商都在用同一套话术，而验证话术是否属实的成本几乎全部转嫁给了用户。对用 AI agent 处理客户代码交付、做外包或自动化服务的档位 C/D 读者，这不是一次性事件，而是需要长期建立的习惯：新工具接入客户项目前，先用抓包工具（如 mitmproxy）验证它实际往哪里传数据。风险在中国市场同样存在——Grok 系服务本身访问就需要额外手段，一旦选择自建本地推理绕开访问限制，反而在某种程度上规避了此次数据外传问题，这是本地化路径的一个意外副作用。

- 项目今年 4 月才创建，3 个月内涨到 7.9 万星、9 千 fork，增长速度远超同类工具的正常曲线，需留意后续维护和安全审计能力是否跟得上（参考本期金矿 1 Grok Build 的教训，爆火期的开源工具尤其需要自己审查依赖和数据流向）

竞品对比
- Claude Design：官方原生方案，体验更顺滑但闭源、云端、锁定 Anthropic 模型且额度紧张；Open Design 用可控性和多模型自由换取了需要自己维护环境的成本

官方链接
- 官网：https://open-design.ai
- GitHub：https://github.com/nexu-io/open-design

深度综述
Open Design 是"闭源官方功能发布几周内必出开源平替"这个已反复验证的 AI 工具生态规律最新一例——从今年上半年一系列"Claude XX 平替""GPT XX 开源版"的爆火轨迹看，这类项目已从"早期信号"进入"中期验证"阶段：只要头部厂商推出一个体验惊艳但捆绑订阅、限制额度的新功能，开源社区几乎必然在几周到一两个月内交出一个自由度更高但需要自己动手配置的替代品。Open Design 把这个规律又验证了一次，且传播速度更快（4 月创建，3 个月破 7.9 万星）。它的护城河不是技术本身（底层仍依赖各家 LLM 能力），而是"集成度"——把 22+ 种 Agent CLI 和 150 套设计系统整合到同一个本地工作流里，这个集成和维护成本才是后来者难以短期复制的部分。风险在于，正如同期 Grok Build 的教训所示，一个几个月内涨了近 8 万星的开源项目，代码审计和依赖链安全很难跟上增长速度，对独立开发者来说"开源"不等于"我看过它的代码"，接入前建议先确认它是否会把设计素材或调用的 API Key传到 Open Design Cloud 之外的地方，尤其是打算用它处理客户项目素材的档位 C/D 读者。

- 定价结构：自出版电子书/纸质书常规定价 + 高级版（含额外内容/装帧，具体定价 [未经验证]）+ 海外版权预付款（一次性，7 笔约¥13.9 万）

- 主要成本：自出版模式下作者需自己承担印刷/物流/翻译协调（如与 @angiewangcreate 合作的《Made in Taiwan》一书提到"第一次自己做印刷和找分销商合作"），利润率因此比交给传统出版社更高，但运营复杂度也更高

## 避坑指南

- 坑 1：AI coding agent 的"隐私开关"可能是摆设。Grok Build 事件里，关闭"Improve the model"开关后，后端接口仍然返回 `trace_upload_enabled: true`，上传通道照常工作——不要因为工具里有一个隐私选项就默认它生效，尤其是处理客户代码或含密钥仓库时，最好的验证方式是自己抓包，而不是读隐私政策页面
- 坑 2：欧洲中性银行（Revolut 等）对"非欧盟股东+跨国持股结构"的出海团队存在冻结风险，且冻结时往往没有邮件通知或明确解释（tibo_maker 案例：10 天无解释，客服只回复"常规检查"）。如果团队有非欧盟籍联合创始人或复杂的跨境持股结构，开户前最好提前咨询能处理这类结构的银行/服务商，并保留资金分散在多个账户的 Plan B，不要把公司全部现金流压在单一账户上
