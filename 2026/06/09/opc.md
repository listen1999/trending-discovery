### 金矿 3：SkillOpt 论文 + koylanai 实操 — markdown skill 文件正在被当作"可训练参数"

来源：@koylanai · 2026-06-08 23:23 · 👁768,999 👍2,276 🔖5,063 · engagement_rate 0.66%（极高，by_views 当日 Top 1 + by_bookmarks 第 2）· type=retweet（RT 自己原文）

核心论点（来自原文整理）
1. **验证闸门**是自编辑 loop 的唯一关键——held-out set + 严格胜出 + 平局拒绝。最优 skill 的接受编辑次数只有 1-4 次。如果"自我改进 agent"接受多数自己提的编辑，那是在"shipping slop"
2. **限制每步编辑量**：4-8 次为最佳，移除该限制性能崩溃。等同于 LLM-as-author loop 的"学习率"
3. **紧凑致胜**：最终 skill 的中位长度仅 **~920 tokens**。多数 skill 文件偏长是因为"长 = 投入"的错觉
4. **跨 harness 可迁移**：Codex 训练出的 skill 移植到 Claude Code，在 SpreadsheetBench 上 **+59.7 分**。程序化知识比承载它的 runtime 更通用
5. **冻结大模型 + 训练上下文**是实用范式：GPT-5.4-nano 加一个 SkillOpt'd skill ≈ frontier 表现。对不训模型的人是最实用的领域适应
6. **验证者本身是瓶颈**：当前 SkillOpt 全部依赖 auto-grader。写作 / 设计 / 策略类开放任务无可用 verifier——谁建出来谁吃下一个阶段

koylanai 自己实测要点（v2.3.0 of Context Engineering Agent Skills repo，跨 composer-2 / claude-opus-4-7 / gpt-5.5 / gemini-3.1-pro）：
- skill 的 description 字段和 body 是两个独立"面"——router 只看 description，agent 只在激活后看 body。两者会悄悄分歧，只有端到端任务测试能抓到
- 看聚合准确率会漏掉单 skill 23-25pp 的差异——per-skill 效应才是真正的杠杆

复制路径

档位 C（工具集成者 / vibe coder）
- 今天 30 分钟可做：对现有 SKILL.md / system prompt 做一次"长度审计"——超过 1,500 tokens 的强制砍到 920 量级
- 本周可做：在自己常用 agent loop 里加一个 held-out validation set（3-5 个真实任务），skill 每次改动都跑一遍，**只接受严格优于上一版的改动**
- 别做的事：不要让 agent 一次性"自我改写整个 skill 文件"——限制 diff 在 4-8 行

档位 B（独立开发者）
- skill 系统在生产中确实有效，但前提是有 verifier。做 ToB 工具 / 客户交付，可以把"我会为你的领域设计/训练 skill"作为差异化卖点；做 ToC 时 verifier 缺失则效果不可控

[关键约束]
- 论文 arXiv 编号原文给到 **2605.23904**，格式异常（arXiv 自 2503 起编号方式不一致）。读者前往 arxiv.org 自行验证再下载 [未经验证]
- "Codex-trained skill → Claude Code +59.7 points"出处仍是该论文本身，本环境未做第三方复现 [未经验证]

深度综述
真正的信号不只是论文，而是 koylanai 这种"实战派"（@sullyai 在职 Agent Research、Personal Brain OS 写作者）做了完整的"理论 ↔ 工程"对照——他特别指出论文里"protected section invariant 防止 fast edit 覆盖 slow lesson"这一条，正好对应他自己 digital-brain-skill 仓库里 voice-guide.md（慢态）vs posts.jsonl（快态）的天然区分，并承认这是他之前没设计的机制。这意味着 SKILL.md 这条产品线在 2026 年正式从"prompt 玩具"进入"工程化资产"阶段。对中国创业者最大的意义是：你过去给客户做的"工作流"以前是 Word 文档 / SOP / 邮件模板——这些资产以后会被改写成 skill 文件被 agent 直接消费，@gregisenberg 同期 60 分钟"AI Native"masterclass 强调的"making your company readable to agents"指的就是这件事。但 5,063 bookmarks 高互动也藏着一个隐性危险：大量读者会停在"我也想搞 skill"这一层，跳过 verifier 这个真正的瓶颈。skill 写出来很容易，但没有 held-out set 验证就只是"另一种 README"。对工具集成者来说，护城河不是"我有多少 skill"，而是"我有 verifier"——这个赛道目前空白，谁先在垂直行业（法律 / 医疗 / 电商运营 / 内容审核）做出可用 verifier，谁就拥有真正的复制壁垒。

## 避坑指南

- **坑 1**：把 Marc Lou $20K MRR 当目标值。他的 $20K 是在 ShipFast（基础设施型产品 + 已有受众）的存量加持下做出的，单立项目的人均数据远不如此。直接拿这个数字定 KPI 会快速进入抑郁式自我比较。
- **坑 2**：把 Levels "8% 成功率"误解为"哪天都能 ship 出爆款"。Levels 自 2014 年起平均每年 5 个新项目，是用密度逼出概率，而不是单次押注。一年只做 1 个产品的人不能用 8% 这个数。
- **坑 3**：以为加 skill 文件就够了。SkillOpt 全文最关键的是 verifier 闸门，没有 held-out set，skill 写得再漂亮也是 random walk。在中文圈 skill / command 已经被包装成"加了就有效"的灵药，与论文原意完全相反。
- **坑 4**：把 indie hacker meetup 的现场吐槽当成"营销 > 产品"的简化结论。真正的论断是分发能力和构建能力都不能省，但 2026 年的边际收益已经倒向分发——这是顺序问题，不是替代问题。
